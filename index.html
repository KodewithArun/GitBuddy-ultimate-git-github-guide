Install Git correctly, configure identity, and use commit and branch conventions that work well in team repositories.

01 — Daily Workflow

The Professional Daily Routine

How a senior developer actually starts, works, and closes out each task in a team environment. This is the foundation everything else builds on.

🌅
Situation: Starting a new task or feature
You've been assigned a ticket. What's the very first thing you do?
▾

Never work on the main branch. Always branch from an up-to-date copy of main.

  • 1
    Sync first. Before anything else, pull the latest changes from the remote so your branch starts fresh.
  • 2
    Create a branch named after the ticket or feature. Use a consistent naming convention your team agrees on.
  • 3
    Do your work in small, logical commits. Don't wait until you're done to commit.
  • 4
    Push regularly so your work is backed up and visible to the team — even before it's done.
# Step 1: Always start by syncing
git switch main
git pull origin main

# Step 2: Create your feature branch
git switch -c feature/user-authentication
# or for a bugfix:
git switch -c bugfix/fix-login-null-error

# Step 3: Do work, commit often
git add src/auth.py
git commit -m "Add JWT token generation"

# Step 4: Push to remote regularly
git push -u origin feature/user-authentication
# after first push, just use:
git push
Branch naming conventions matter. Common formats: feature/TICKET-123-short-description, bugfix/issue-description, hotfix/critical-fix-name. Pick one and stick to it as a team.
📝
Situation: Writing commits like a senior developer
Your commits are permanent documentation. Bad messages haunt teams for years.
▾
❌ Bad✅ Professional
fixFix null pointer exception in payment validator
updateUpdate session timeout from 30min to 60min per UX request
WIPAdd draft of OAuth integration (tests pending)
asdfghRemove deprecated API v1 endpoints
changesRefactor user model to support multiple email addresses

The professional format for multi-line commits:

# Open editor for multi-line commit:
git commit

# Your editor opens — write this structure:

Add JWT authentication for API endpoints
(blank line here)
Replaced session-based auth with JWT because mobile clients
don't handle cookies reliably. Tokens expire in 24h.
Refresh token stored in httpOnly cookie for security.
(blank line)
Closes #42
Co-authored-by: Priya Sharma <priya@company.com>
Rule: Subject line max 50 characters. Use imperative mood ("Add", "Fix", "Remove", not "Added", "Fixed", "Removed"). Explain WHY in the body, not just WHAT.
🔄
Situation: Syncing when the team keeps pushing to main
You've been working on your branch for 3 days. Main has moved on. How do you catch up?
▾

This happens constantly in team environments. The longer you wait, the worse the conflicts.

  • 1
    Fetch the latest changes from remote (doesn't modify your working files)
  • 2
    Rebase your branch on top of the updated main — this replays your commits cleanly
  • 3
    Force-push (safely) since rebase rewrites your branch history
# You're on feature/my-feature
git fetch origin

# Rebase your work on top of the latest main
git rebase origin/main

# If conflicts arise, resolve them, then:
git add <resolved-files>
git rebase --continue

# Force push (only safe because it's YOUR personal branch)
git push --force-with-lease
--force-with-lease vs --force: Always use --force-with-lease. It refuses to overwrite if someone else has pushed to your branch since your last fetch. Much safer than bare --force.
⚠️ Never force-push to main or any shared branch. Only force-push to your own personal feature branches.
02 — Branching Strategy

Branching Like a Professional

The branch model your team uses determines how smoothly code flows from development to production. Here are the three most common setups.

🌿
Situation: Understanding which workflow your team uses
GitHub Flow vs Git Flow vs Trunk-based — when to use which
▾

GitHub Flow — most common for web apps and startups

main → feature/login → PR → main → deploy

Rule: main is always deployable. All work happens in branches. PR → Review → Merge → Deploy immediately.

Git Flow — for versioned software with scheduled releases

main ↑ release/v2.0 ↑ develop ← feature/*
hotfix/* → main + develop

Use when: mobile apps, desktop software, open-source libraries with versioned releases.

WorkflowBranchesBest forRelease cadence
GitHub Flowmain + feature branchesWeb apps, SaaSMultiple times/day
Git Flowmain, develop, feature, release, hotfixMobile, versioned softwareScheduled releases
Trunk-basedmain only (tiny branches <1 day)Google/Facebook scaleContinuous
🔀
Situation: Merge vs Rebase — making the right choice
Both integrate branches. The choice affects your history's readability forever.
▾
# MERGE — preserves true history, creates a merge commit
git switch main
git merge feature/user-auth
# History: A - B - C - M  (M has two parents)
#                  \   /
#                   D-E

# REBASE — rewrites history to be linear, no merge commit
git switch feature/user-auth
git rebase main
# History: A - B - C - D' - E'  (clean line)
Use Merge when...Use Rebase when...
Merging a finished feature PR into mainUpdating your feature branch with latest main
You want to preserve the full contextCleaning up local commits before opening a PR
Working with shared/public branchesWorking on private, not-yet-pushed branches
The branch divergence is meaningfulYou want a linear, readable history
The Golden Rule: NEVER rebase commits that are already on the shared remote and other people may have pulled. You will rewrite history out from under them.
⚔️
Situation: Merge conflict — two developers changed the same lines
Conflicts are normal. The professional response is calm and systematic.
▾
  • 1
    Attempt the merge — Git will tell you which files conflict
  • 2
    Open each conflicted file — find the conflict markers
  • 3
    Decide: keep yours, keep theirs, or combine intelligently
  • 4
    Remove all conflict markers, save the file
  • 5
    Stage the resolved file and complete the merge
# Conflict markers look like this in your file:
<<<<<<< HEAD
DEBUG = False   # your version (current branch)
=======
DEBUG = True    # incoming version (being merged)
>>>>>>> feature/dev-settings

# After resolving (edit the file to correct state):
git add config.py
git commit    # completes the merge

# To abort the merge entirely and go back:
git merge --abort

# Use VS Code's merge editor (excellent for conflicts):
code .  # conflicts highlighted, Accept/Both/Ignore buttons
Best practice: Before a merge, talk to the other developer. Understanding WHY they made their change makes the resolution obvious and correct.
03 — Team Collaboration

Working Professionally in a Team

The PR workflow, code review culture, and handling production emergencies. This is where Git skills translate into team credibility.

🔁
Situation: Opening a Pull Request the right way
Your PR is your professional pitch. A bad PR wastes reviewers' time and slows the team.
▾
# 1. Make sure your branch is up to date before opening PR
git fetch origin
git rebase origin/main

# 2. Push your branch
git push -u origin feature/user-auth

# 3. Open PR on GitHub — use this description template:
## What this PR does
Adds JWT-based authentication to all API endpoints.

## Why
Session auth breaks with mobile clients. Fixes #42.

## Changes made
- Added JWTMiddleware to express app
- Token expiry: 24h access, 7d refresh
- Updated all protected routes

## Testing
- All existing tests pass
- Added 12 new tests in auth.test.js

## Screenshots (if UI change)
[attach screenshot]
  • Keep PRs small — under 400 lines ideally. Big PRs get rubber-stamped or take days to review.
  • Open a "Draft PR" early if you want feedback before it's done.
  • Assign reviewers — don't just open it and hope someone notices.
  • Link the issue with Closes #42 — it auto-closes when merged.
  • Add a self-review first — read your own diff before asking others to.
🔍
Situation: Doing code review professionally
How you review code reflects your seniority. Aggressive reviews kill team morale. Rubber-stamping kills code quality.
▾

As a reviewer:

  • DO Ask questions: "Could we use a constant here for readability?"
  • DO Be specific: "Line 42: null check missing if user is undefined"
  • DO Distinguish blocking vs non-blocking: prefix with "nit:" for minor style issues
  • DO Approve when it's good enough — don't chase perfection
  • NEVER "This is wrong" without explanation
  • NEVER Rewrite the PR yourself in comments
  • NEVER Let a PR sit for more than 24 hours unreviewed

As the PR author:

  • Respond to every comment, even if just "Fixed" or "Agreed, keeping as-is because..."
  • Don't argue style — either agree or escalate to team discussion
  • When you push new commits in response to review, comment "Addressed in abc1234"
🚨
Situation: Critical bug in production right now
Production is down at 2am. You need to fix and deploy in under 30 minutes. Exact steps.
▾
# 1. Drop everything — switch to main and pull LATEST production code
git switch main
git pull origin main

# 2. Create a hotfix branch from main (NOT develop)
git switch -c hotfix/fix-payment-crash

# 3. Fix the bug. Keep it MINIMAL — only fix the crash, nothing else
# ... make your fix ...
git add payment.py
git commit -m "Fix divide-by-zero crash in payment calculator

Amount field was not validated before division.
Added null/zero check. Fixes #PROD-2847"

# 4. Push and open PR — or if you have emergency merge rights, direct merge
git push -u origin hotfix/fix-payment-crash

# 5. After merging to main AND tagging a release — also merge to develop
# So the fix is also in the next scheduled release
git switch develop
git merge hotfix/fix-payment-crash

# 6. Tag the hotfix release
git tag -a v1.2.1 -m "Hotfix: payment crash fix"
git push origin v1.2.1
Hotfix discipline: Only fix the exact bug. Resist the urge to "clean up" other things while you're there. Every extra change is another risk during a production incident.
04 — Recovery & Safety

Undoing Mistakes Without Panic

Every developer breaks things. Seniors don't panic because they know exactly how to recover. Here's your complete recovery toolkit.

↩️
Situation: I committed something wrong — how do I undo it?
The answer depends on whether you've pushed yet. Choose the right tool.
▾
ScenarioCommandSafe on shared branch?
Didn't commit yet, want to unstage git restore --staged <file> Yes
Didn't commit, want to discard file changes git restore <file> Yes
Last commit: fix message only git commit --amend -m "New message" Not pushed
Last commit: undo, keep changes staged git reset --soft HEAD~1 Not pushed
Last commit: undo, unstage changes git reset HEAD~1 Not pushed
Commit already pushed to shared branch git revert <hash> Yes — safe
Entire last commit: destroy it completely git reset --hard HEAD~1 Never on shared 
# The safe undo for anything already pushed to shared branches:
git revert a3f2b1c
# Creates a NEW commit that undoes the bad commit
# History is preserved. Safe for teammates.

# Accidentally committed to main directly?
# (Before pushing)
git reset --soft HEAD~1     # undo commit, keep changes staged
git switch -c feature/my-branch   # now put it on a branch
git commit -m "Proper commit message"
📦
Situation: Need to switch context but have unfinished work
"Urgent bug just came in but I'm mid-feature and can't commit yet." Use git stash.
▾
# Save your work-in-progress (WIP) temporarily
git stash push -u -m "WIP: implementing OAuth, half done"
# -u includes untracked new files

# Now your working directory is clean — handle the urgent bug
git switch main
git switch -c hotfix/urgent-fix
# ... fix, commit, push, open PR ...

# Come back to your stashed work
git switch feature/oauth
git stash list
# stash@{0}: WIP: implementing OAuth, half done

git stash pop  # apply latest stash AND remove it from stash list
# or
git stash apply stash@{0}  # apply but keep in stash list

# Clean up stashes you no longer need
git stash drop stash@{0}
git stash clear  # delete ALL stashes
Stashes are local only — not pushed to GitHub. Don't stash things and then forget about them. Name your stashes clearly.
🆘
Situation: I accidentally deleted a branch / lost commits
git reflog is your emergency time machine. Almost nothing is truly lost.
▾
# Show everything HEAD has ever pointed to
git reflog
# Output:
a3f2b1c HEAD@{0}: reset: moving to HEAD~2
b4c2d1e HEAD@{1}: commit: Add payment module
c5d3e2f HEAD@{2}: commit: Fix session handling
d6e4f3a HEAD@{3}: checkout: moving from feature/x to main

# Recover the "lost" commit by creating a branch at it
git switch -c recovery-branch b4c2d1e

# Or reset your current branch to that point
git reset --hard b4c2d1e

# Accidentally deleted a branch?
# Find the last commit that was on that branch in reflog, then:
git switch -c restored-branch <hash>
Git keeps deleted commits in its database for ~30 days before garbage collection. The reflog is your window into that. Use it calmly — you're almost certainly not actually out of luck.
05 — Advanced Power Commands

Advanced Git — Power User Level

These are the commands that differentiate mid-level from senior developers. Use them with precision.

✂️
Situation: Your commit history is a mess before opening a PR
WIP commits, typo fixes, "argh" messages — clean it all up with interactive rebase.
▾
# Your ugly history before clean-up:
wip
fix typo
fix typo again
maybe working now?
Add user authentication

# Interactively edit the last 5 commits:
git rebase -i HEAD~5

# Editor opens — change pick to squash/fixup/reword:
pick  a3f2b1c Add user authentication
squash b4c2d1e maybe working now?    # merge into above
fixup  c5d3e2f fix typo again         # merge, discard message
fixup  d6e4f3a fix typo              # merge, discard message
fixup  e7f5a4b wip                   # merge, discard message

# Result: ONE clean commit "Add user authentication"
# Then force-push your personal branch:
git push --force-with-lease

Interactive rebase commands:

CommandWhat it does
pick / pKeep commit as-is
reword / rKeep commit, edit message
squash / sMerge into previous, combine messages
fixup / fMerge into previous, discard this message
drop / dDelete this commit entirely
edit / ePause here to amend the commit
🍒
Situation: A fix on a feature branch is urgently needed on main
Cherry-pick copies a specific commit to another branch without merging the whole branch.
▾
# Scenario: bug fix committed to feature/payments
# You need ONLY that fix on main, not the whole feature

# 1. Find the commit hash you want
git log feature/payments --oneline
# b4c2d1e Fix divide-by-zero in price calculation  <-- this one
# a3f2b1c Add new payment gateway (not ready yet)

# 2. Switch to main and cherry-pick that exact commit
git switch main
git cherry-pick b4c2d1e

# That commit is now on main as a new commit (new hash)
# The feature branch is unaffected

# Cherry-pick multiple commits:
git cherry-pick a3f2b1c b4c2d1e c5d3e2f

# Cherry-pick a range:
git cherry-pick a3f2b1c..c5d3e2f
🏷️
Situation: Releasing a version to production
Tags mark releases permanently. Every production deployment should be tagged.
▾
# Semantic versioning: MAJOR.MINOR.PATCH
# PATCH: bug fix (1.0.0 → 1.0.1)
# MINOR: new feature, backwards compatible (1.0.0 → 1.1.0)
# MAJOR: breaking change (1.0.0 → 2.0.0)

# Create an annotated tag (always use annotated, not lightweight)
git tag -a v1.2.0 -m "Release v1.2.0: Add payment module, fix auth bug"

# Tags are NOT pushed by default — you must push explicitly
git push origin v1.2.0
git push origin --tags  # push ALL tags

# List all tags
git tag
git tag -l "v1.*"  # filter by pattern

# Go back to exactly how the code was at v1.1.0
git checkout v1.1.0  # detached HEAD — use to inspect only

# Delete a tag (use carefully — others may have pulled it)
git tag -d v1.2.0
git push origin --delete v1.2.0

07 — Team Rules

The Unbreakable Rules of Professional Git

These aren't suggestions. Violating them in a team environment causes real damage.

  • NEVER Commit secrets to Git. API keys, passwords, .env files. Even if deleted later — history is forever.
  • NEVER Force-push to main, develop, or any shared branch. You rewrite history for everyone.
  • NEVER Work directly on main. Every change goes through a branch and a PR.
  • NEVER Rebase commits that are already on a shared remote and pulled by others.
  • ALWAYS Pull before you start working. Don't start the day on a stale codebase.
  • ALWAYS Write meaningful commit messages. Your message is documentation read by future humans.
  • ALWAYS Keep branches short-lived. A branch open for 2 weeks is a conflict waiting to happen.
  • ALWAYS Tag every production release. Semantic versioning, annotated tags, pushed to remote.
  • ALWAYS Commit node_modules / .venv / build artifacts to .gitignore, never to the repo.
  • CAUTION git reset --hard destroys uncommitted work. Verify twice before using it.
  • CAUTION git push --force is almost always wrong. Use --force-with-lease if you must force-push your own branch.
